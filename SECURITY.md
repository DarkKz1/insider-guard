# Безопасность Insider Guard

> «Наш инструмент безопасности — сам безопасен.»
> Аудит проведён по методологиям **OWASP Top-10 (2021)** и **STRIDE**.
> Трек *AI Shield*, АФМ AI Hackathon 2026 — прямой ответ на критерий **#3 (Информационная безопасность + этические нормы)**.

## 1. Резюме

Insider Guard — это UEBA-движок обнаружения инсайдерских угроз (Node.js + Express,
без внешних SaaS-зависимостей для самой AI-модели). Поскольку продукт сам является
инструментом ИБ, к нему применён аудит уровня OWASP/STRIDE. Ниже — модель угроз,
реализованные меры и результаты фиксов.

**Статус аудита:** уязвимости устранены, `npm audit` — **0 vulnerabilities**,
контракт-тест (включая `/api/ingest`) — **PASS**.

## 2. Модель угроз (STRIDE)

Граница доверия: HTTP-поверхность `/api/*` (Express) ← аналитик SOC / загружаемый лог.
Активы: загруженные журналы событий, вычисленные инциденты, снапшот хранилища,
клиентский API-ключ Anthropic (опционально).

| STRIDE | Угроза для Insider Guard | Мера в продукте |
|---|---|---|
| **S**poofing (подмена) | Поддельный источник запроса | Однооригинная модель развёртывания (on-prem / single-origin); ключи доступа к хранилищу (Supabase) только из ENV; статика и API на одном origin. |
| **T**ampering (искажение) | Вредоносный/искажённый загружаемый файл, инъекция в парсер | Строгая валидация формата (CSV/JSON/JSONL/NDJSON/TXT), `fileFilter` по расширению **и** MIME; нормализация и проверка обязательных полей; санитизация имени датасета (удаление управляющих символов, ограничение длины). Парсинг без `eval`/динамического кода. |
| **R**epudiation (отказуемость) | «Я этого не загружал/не решал» | Каждый инцидент — машинный **черновик**, помечен «требует проверки аналитиком» (human-in-the-loop); инциденты имеют стабильные ID и метку времени формирования; решение принимает человек, не модель. |
| **I**nformation Disclosure (утечка) | Утечка ПДн / секретов | **0 реальных ПДн** — весь корпус синтетический (ИИН/ФИО/IP вымышлены); секреты только из ENV (нет хардкода); клиентский API-ключ Anthropic **никогда не сохраняется** на сервере (живёт только в рамках одного запроса). |
| **D**enial of Service (отказ в обслуживании) | Перегрузка большим payload / множеством файлов / утечка памяти multer | multer обновлён до **2.x** (закрыты CVE 1.x); лимиты: размер файла 25 МБ, ровно 1 файл, ≤5 полей, ≤10 частей; лимит JSON-тела 32 МБ; потолок 200 000 событий на запрос; urlencoded ≤256 КБ / ≤100 параметров. |
| **E**levation of Privilege (повышение прав) | Выход за пределы хранилища, выполнение кода | Нет path traversal: пути к хранилищу берутся **только** из ENV, пользовательский ввод не участвует в построении путей; загрузки в памяти (`memoryStorage`) — нет временных файлов на диске; нет `child_process`/`exec`/`eval`. |

## 4. Реализованные фиксы (этот аудит)

1. **multer 1.x → 2.2.0** (`package.json`, `package-lock.json`). Закрыты известные CVE
   класса DoS (утечка памяти / необработанное исключение) из 1.x, на которые указывал Vercel.
   API multer 2.x для `memoryStorage()` / `single('file')` / `req.file.buffer` совместим —
   маршрут `/api/ingest` не сломан (подтверждено контракт-тестом).
2. **Лимиты загрузки** (`server/routes/ingest.routes.js`): `fileSize` 25 МБ, `files: 1`,
   `fields: 5`, `parts: 10`, `headerPairs: 100`.
3. **Валидация типа файла**: `fileFilter` проверяет расширение (CSV/JSON/JSONL/NDJSON/TXT)
   и MIME-тип; недопустимый файл → **415**, слишком большой → **413** (чистый JSON, не 500).
4. **Лимит событий** в JSON-пути ingest: ≤200 000 событий → иначе **413**.
5. **Санитизация имени датасета**: удаление управляющих символов `\x00–\x1f\x7f`,
   обрезка до 200 символов.
6. **Ужесточение body-parser** (`server/server.js`): `express.json` 64 МБ → 32 МБ;
   `express.urlencoded` с `limit: 256kb`, `parameterLimit: 100`.
7. **Валидация клиентского API-ключа** (`server/routes/report.routes.js`): проверка формата
   `^sk-ant-[A-Za-z0-9_-]{20,200}$` перед отправкой в Anthropic; некорректный ключ →
   graceful fallback на offline mock-отчёт (ключ не пересылается и не сохраняется).

## 5. Этика и соответствие (критерий #3 — этические нормы)

- **0 реальных персональных данных.** Весь демонстрационный корпус — синтетический
  (ИИН, ФИО, IP-адреса вымышлены). Продукт не обрабатывает реальные ПДн в прототипе.
- **Human-in-the-loop / EU AI Act.** Система высокого риска (HR/безопасность) не принимает
  автономных решений: каждый инцидент-отчёт — **черновик**, явно помеченный «требует проверки
  аналитиком». Финальное решение принимает человек. Это соответствует требованию
  человеческого надзора (Art. 14 EU AI Act) и принципу объяснимости.
- **Объяснимость.** Каждый risk-score раскладывается через SHAP-вклады (сумма вкладов = score),
  граф доступа и перечень триггеров — аналитик видит *почему* сработал детектор, а не «чёрный ящик».
- **AI-модель независима от внешних сервисов.** Детект-движок UEBA (baseline-per-user,
  скоринг, типологии) работает локально, без облачных AI-API. LLM (Anthropic) используется
  **только опционально** и **только** для генерации текста IR-отчёта по клиентскому ключу;
  ядро обнаружения от него не зависит.
- **Ключ Anthropic — клиентский и опциональный.** Передаётся в теле запроса для одного вызова,
  не логируется и не сохраняется на сервере. Без ключа продукт полностью функционален
  (offline mock-отчёт).
- **On-prem / суверенность данных.** Разворачивается внутри периметра организации; логи
  событий не покидают инфраструктуру (хранилище in-memory + диск, опционально Supabase
  под контролем владельца).

## 6. Известные ограничения (честный scope MVP)

- **Аутентификация/авторизация на уровне приложения отсутствует** — MVP рассчитан на
  развёртывание за периметром организации (on-prem, доверенная сеть SOC). Для продакшена
  следующий шаг — SSO/OIDC + RBAC перед `/api/*`.
- **CORS открыт (`*`)** для удобства демо. В продакшене — заменить на конкретный origin.
- **Rate-limiting** не включён в MVP; рекомендуется reverse-proxy (nginx) или
  `express-rate-limit` при публичном развёртывании.
