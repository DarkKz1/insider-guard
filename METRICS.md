# METRICS — Insider Guard

> Документ для команды и для подготовки к Q&A жюри. Цель: чтобы все на сцене
> называли ОДНИ И ТЕ ЖЕ числа и формулировали их ЧЕСТНО. Нечестная метрика —
> главная «ловушка жюри» на ИБ/AI-треке.

---

## 0. ВАЖНО: какие числа идут на сцену

**Метрики, которые видит жюри на демо, считаются на вкладке «Валидация» в
`index.html` из ВСТРОЕННОГО в страницу массива `CASES` — НЕ из server-сида.**

У demo-вкладки и у бэкенд-сида **разные корпуса** (разный размер, разный набор
кейсов), поэтому их числа НЕ совпадают и совпадать не должны. Не смешивать.

- **Числа для сцены → берёт и фиксирует QA** из вкладки «Валидация» (`CASES` в
  `index.html`). Это источник правды для презентации.
- **Server-метрики (раздел 3 ниже) — справочные**, для контракт-теста и для
  ответа «а как считает бэкенд». На слайды их НЕ выносим, чтобы не было двух
  разных цифр в одном выступлении.

> Backend-инженер НЕ трогает `index.html`. Фактические demo-числа подтверждает
> QA после прогона вкладки. Раздел 1 (правила формулировки) применим к ЛЮБОМУ
> источнику.

## 1. ПРАВИЛА ЧЕСТНОЙ ФОРМУЛИРОВКИ (обязательно, из плана §11)

Говорить так:

- «**N инцидентов / 8 поведенческих типологий покрыто / benign-контроли НЕ
  помечены**» — типологии: mass_exfil, lateral, priv_escalation, off_hours,
  staging_exfil, compromise, broad_access, covert_channel.
- Показывать **recall@topN** и **AUPRC** (устойчивы к дисбалансу классов).
- Явно называть benign hard-negatives (etl_service, planned_audit), которые
  система НЕ пометила → доказательство низкого false-positive.

НИКОГДА не говорить так:

- ❌ «**8/8 атак**» — звучит как заявление об идеальной модели на крошечной
  выборке; жюри сразу спросит про размер корпуса и переобучение.
- ❌ Гнать **accuracy** как главную метрику.

**Accuracy-trap (заготовить ответ на 15 сек):** при дисбалансе 8 угроз на ~900
benign модель-пустышка, которая всегда отвечает «не угроза», даёт
**accuracy ≈ 99.9%** и при этом ловит **ноль** угроз (recall = 0). Поэтому
accuracy здесь бессмысленна — мы смотрим precision / recall@topN / AUPRC.

**Атрибуция кейса:** «авторизованный доступ, инсайдер ЛИБО компрометация» —
никогда «доказанный инсайдер».

**Имя модели:** Claude используется ТОЛЬКО для IR-нарратива/отчёта, НИКОГДА не
считает score. Реально вызывается `claude-opus-4-8` (`server/report.js:69`);
эндпоинт `/api/report/:id` теперь возвращает тот же id (`report.routes.js:19`).

## 2. Что значат метрики (шпаргалка для Q&A)

| Метрика | Смысл | Почему её показываем |
|---|---|---|
| **recall@topN** | доля реальных угроз, попавших в топ-N очереди (N = число угроз) | аналитик смотрит топ очереди — это и есть его рабочая метрика |
| **AUPRC** | площадь под precision-recall | честная при сильном дисбалансе классов |
| **precision** | доля верных среди помеченных | «сколько ложных тревог получит SOC» |
| **recall** | доля пойманных из всех угроз | «сколько угроз пропустили» |
| **naive-DLP reduction** | насколько меньше ложных тревог, чем у порогового DLP на ТОМ ЖЕ корпусе | показывает ценность UEBA над «DLP по объёму» |
| ~~accuracy~~ | (trap) | НЕ выносить как главную — см. §1 |

## 3. SERVER-метрики (СПРАВКА — НЕ для сцены)

Источник: server-сид (`server/seed/*`), эндпоинт `GET /api/metrics?threshold=55`.
Сид детерминирован (UTC-anchored), числа воспроизводимы. Получено `2026-06-24`
прогоном `npm start` + curl. Это НЕ числа demo-вкладки.

Корпус сида:
- events: **12 344**, users: **40**, days: **30**, datasets: **18 ресурсов / 16 хостов**
- размеченных user-day: **905** (illicit **8**, benign **897**)
- инцидентов в очереди: **55** (crit 2 / high 6 / medium 24 / low 23)
- **8 малицизных типологий** + **2 benign hard-negatives** (etl_service, planned_audit)

Качество @ threshold = 55:

| Метрика | Значение |
|---|---|
| confusion | TP 8 / FP 0 / FN 0 / TN 897 |
| precision | 1.0 |
| recall | 1.0 |
| f1 | 1.0 |
| recall@topN | 100% |
| AUPRC | 1.0 |
| alertsShown | 8 |
| naive-DLP | TP 8 / FP 46 / alerts 54 / **reduction 100%** |

> Замечание по зрелости: на сцену precision=recall=AUPRC=1.0 выносить осторожно —
> «идеальные 1.0» на синтетике провоцируют вопрос про переобучение. Корректная
> подача: «на нашем размеченном корпусе все 8 типологий поднялись в топ очереди
> выше всех benign; naive-DLP на том же корпусе дал 46 ложных тревог — мы их
> убрали». Числа demo-вкладки (раздел 0) могут отличаться — берём их у QA.

### Воспроизведение

```
npm start            # поднимает http://localhost:3000 (in-memory+disk store)
npm run test:contract
curl -s "http://localhost:3000/api/metrics?threshold=55"
```

`threshold` клампится 0–100 на бэкенде → what-if слайдер уже подключён к готовому API.
