# AI Shield — Insider Guard

Прототип для трека **AI Shield** (`id: ai-shield`), AFM AI Hackathon 2026 (Алматы, 24–25 июня).
Трек AI Shield = **защита данных / антихакинг**.

> «AI-щит данных»: поток access/audit-логов (кто → к какому ресурсу → с какого хоста) →
> детект **инсайдерской угрозы и компрометации аккаунта** по отклонению от поведенческого
> baseline пользователя (UEBA) → **человекочитаемое объяснение, почему помечено** +
> рекомендация реагирования (IR-плейбук) + **приоритизированная очередь** инцидентов для SOC.

---

## Что это

Дашборд SOC / инцидент-аналитика. На вход — **access/audit-логи** (журналы доступа к БД,
аутентификации хостов, IAM-grant'ов, исходящих выгрузок). На выход — **приоритизированная
очередь инцидентов** с risk-score 0–100, **графом доступа** (пользователь → ресурс → хост),
**объяснением по факторам (SHAP-подобная декомпозиция)**, рекомендацией реагирования и кнопкой
**«сформировать черновик инцидент-отчёта (IR report)»**.

Ключевая идея: продукт ловит угрозу, которую **периметровый DLP/SIEM не видит** — у инсайдера
доступ **легальный**. Поэтому детект строится не на сигнатурах атаки, а на **отклонении от
персонального baseline самого пользователя** (UEBA): объём, время, широта доступа, привилегии,
гео/устройство. Инструмент **не заменяет** аналитика, а **сортирует его работу** (самое опасное
наверх) и **объясняет каждое решение**, чтобы человек мог проверить.

> **Это НЕ AML.** Прошлая версия прототипа (детект отмывания/транзакций) сохранена как бэкап
> `index_aml_backup_jun20.html`. Текущий продукт — кибербез/защита данных, никаких транзакций
> и финмониторинга.

## Якорь (зачем это нужно сейчас)

**Утечка 16,3 млн записей РК (июнь 2025) — это был инсайдер с авторизованным доступом, а не взлом
периметра.** Файрвол, антивирус и DLP такое не ловят: запрос к базе выглядит «легально». Единственное,
что выдаёт инсайдера, — **аномалия его собственного поведения**: выгрузка в 80× больше нормы, ночью,
по таблице ПДн. Insider Guard построен ровно вокруг этого слепка (кейс `INC-7042`).

## Как открыть

```
open index.html
```

Просто **двойной клик по `index.html`** или перетащить в браузер. Один самодостаточный файл.
React + Tailwind + Babel грузятся с CDN — нужен интернет **только при первом открытии** (для
загрузки этих 3 библиотек). Вся логика детекта и все данные — **инлайн, оффлайн, без сервера и
без `npm install`**. Детект работает полностью локально.

## Демо-сценарий (4 шага)

1. **Очередь.** Слева — 9 инцидентов, отсортированных по risk-score. Сверху критические **P1**:
   массовая выгрузка ПДн (`INC-7042`, слепок 16,3 млн) и боковое перемещение к БД физлиц
   (`INC-7088`). Покажите жюри: «SOC работает сверху вниз, не тонет в шуме».
2. **Объяснимость.** Кликните верхний инцидент → справа панель **«Почему помечено»**: каждый триггер
   (VOLUME_ANOMALY, LATERAL_MOVEMENT, PRIV_ESCALATION…) с его **вкладом в score** (SHAP-подобно) и
   человеческим пояснением относительно baseline пользователя. Это решающий сигнал — прозрачность модели.
3. **Граф доступа (вау-момент).** Кликните `INC-7088` → граф рисует цепочку бокового перемещения
   `АРМ → промежуточный хост → сервер приложений → сервер БД → база физлиц` красными рёбрами.
   Кликните `INC-7240` → видно веер из 12 БД вне профиля (scatter-gather разведки). Граф объясняет
   почерк визуально.
4. **Черновик IR-отчёта.** Внизу нажмите **«Сформировать черновик»** → движок собирает инцидент-отчёт
   (резюме, хронология, затронутые ресурсы и объём, индикаторы, рекомендации) из триггеров (RU).
   Экономит SOC-аналитику бумажную работу.

> **Два контрольных negative-инцидента** доказывают калибровку (система не красит «всё подряд»):
> **`INC-9001` (ETL/бэкап сервис-аккаунт)** — большой ночной объём (90k строк), но это его
> established baseline → подавлен; **`INC-9002` (плановая проверка аудитора)** — повышенный объём
> ×3, но в рабочее время по своим таблицам → подавлен. Именно baseline-контекст бьёт наивный
> объёмный порог «прочитал > N строк».

## Опционально: реальный Claude вместо mock

В блоке IR-отчёта есть поле для Anthropic API-ключа. Без ключа черновик генерится
**детерминированным mock-шаблоном** (по умолчанию, работает оффлайн). С ключом — реальный вызов
`claude-sonnet-4-6` для более естественного нарратива. Mock — дефолт, демо не зависит от сети/ключа.

## Синтетические данные: 9 инцидентов (KZ-колорит)

9 инцидентов, каждый — подграф доступа `{пользователи, ресурсы (БД/таблицы ПДн), хосты}` с известной
типологией (ground-truth для метрик). **7 illicit + 2 legit-контроль.** На каждый аккаунт задан
baseline `{avg_rows_per_day, work_hours, known_tables}` — норма поведения, относительно которой
считается отклонение.

Таблица — **пример прогона** (jun20). Score не зашит в данные, а **генерится движком** (`analyze()`:
взвешенная сумма триггеров + насыщение + контекст-снижение для established-аккаунтов), поэтому числа
могут чуть сдвинуться при правке весов/порогов.

| Инцидент | Типология | KZ-сюжет | Класс |
|----------|-----------|----------|-------|
| INC-7042 | mass_exfil | Аналитик выгружает 80k строк ПДн в 02:14 при baseline 50/день (×80) — слепок 16,3 млн | illicit (P1) |
| INC-7088 | lateral | Скомпрометированный аккаунт прыгает по 4 хостам до БД физлиц | illicit (P1) |
| INC-7110 | priv_escalation | Джуниор получает admin-grant вне заявок → читает зарплату/санкции | illicit |
| INC-7155 | off_hours | Всплеск SELECT по налоговой базе в воскресенье (вне work_hours) | illicit |
| INC-7177 | staging_exfil | Прочитал 60k → выгрузил 58k наружу (≈1:1, near-zero оседание) | illicit |
| INC-7203 | compromise | Impossible travel: Астана → зарубеж за 8 мин + новое устройство → доступ к ПДн | illicit |
| INC-7240 | broad_access | Поддержка трогает 12 новых БД вне профиля за час (scatter-gather) | illicit |
| INC-9001 | etl_service | ETL/бэкап сервис-аккаунт: плановый ночной объём 90k → **подавлен** | legit (контроль) |
| INC-9002 | planned_audit | Аудитор: повышенный объём ×3 в рабочее время по своим таблицам → **подавлен** | legit (контроль) |

> **Контрольные negative (precision):** `INC-9001` и `INC-9002` имеют объём, который наивный
> детектор пометил бы (отсюда индустриальные ~95% ложных алертов). Но baseline-контекст
> (established / ночной job / рабочее время / свои таблицы, без жёстких крит-индикаторов) снижает
> риск (×0.45) и держит их ниже порога тревоги. Это демонстрирует precision, а не только recall.

Легенда: гос-орган / среда АФМ РК, базы ПДн по модели утечки 16,3 млн (ФИО / ИИН / адрес / телефон).
Поля события: `{user, role, resource, db, host, ip, geo, action, rows, ts, channel}`.
**Всё синтетическое: ИИН / ФИО / IP вымышлены, никаких реальных персональных данных.**

## Архитектура детекта (mock-AI, детерминированный · UEBA)

Движок построен на каркасе «архитектуры-победителя» из AML-версии (граф + табличные триггеры +
scoring + SHAP + AUPRC) — **сам каркас переиспользован**, а доменная логика переписана с
транзакций на access-логи. `buildGraph` (агрегация по рёбрам: было `amount` → стало `rows`),
DFS-обход (был поиск цикла A→B→C→A → стал поиск пути бокового перемещения через ≥3 хоста),
взвешенный scoring, SHAP-декомпозиция (largest-remainder), AUPRC и наивный baseline — всё это
осталось от движка AML и переиспользовано.

| Слой | Что делает | Реализация |
|------|-----------|------------|
| **Граф доступа** | боковое перемещение, веерный доступ к множеству БД, путь к данным | adjacency + DFS-поиск пути через ≥3 хоста (networkx-эквивалент в JS) |
| **Поведенческие триггеры** | отклонение от baseline: объём / время / широта / привилегии / гео | пороговые эвристики **относительно персонального baseline пользователя** |
| **Scoring** | risk-score 0–100 + приоритет P1–P4 | взвешенная сумма триггеров с насыщением (sigmoid-подобно) + контекст-снижение established |
| **Объяснимость** | вклад каждого фактора | SHAP-подобная additive-декомпозиция (largest-remainder, сумма === score) |
| **Метрики** | честная оценка качества на дисбалансе | AUPRC + recall@top-N + confusion-matrix под what-if порог (НЕ accuracy) |
| **LLM-слой** | черновик IR-отчёта | mock-шаблон по умолчанию / реальный Claude по ключу |

**8 кибер-триггеров на baseline-отклонениях:**

1. **VOLUME_ANOMALY** — объём чтения/выгрузки ≫ персонального baseline (×80 либо абсолютный пол 50k при норме <500/день).
2. **LATERAL_MOVEMENT** — граф-путь через ≥3 хоста до ресурса (обход сегментации, почерк скомпрометированного аккаунта).
3. **BROAD_ACCESS** — доступ к ≥5 ресурсам вне `known_tables` (scatter-gather разведки). Слабая форма — **SENSITIVE_ACCESS** (1–4 ресурса вне профиля).
4. **BULK_EXFIL** — молниеносная выгрузка наружу (EXPORT/download) после массового чтения.
5. **OFF_HOURS_VELOCITY** — операции вне `work_hours` / в выходной на фоне всплеска активности.
6. **PRIV_ESCALATION** — GRANT/SUDO/смена роли вне процесса заявок.
7. **COMPROMISE_INDICATORS** — новый гео + новое устройство + impossible travel в одном окне.
8. **STAGING_EXFIL** — прочитано ≈ выгружено (near-zero оседание, passthrough-примитив под эксфильтрацию).

Плюс служебный **COVERT_CHANNEL** (выгрузка в облако/мессенджер/личную почту, ×2 если канал ранее
помечен) и **CLEAN** (когда аномалий нет — ложная тревога не формируется). Триггеры на одном узле
взаимоисключаются (VOLUME ⊥ BROAD ⊥ SENSITIVE), чтобы один всплеск не считался дважды.
Привязка к MITRE ATT&CK: T1530, TA0008/T1021, TA0004/T1098, T1078, T1074→T1048, T1213.

## Маппинг на критерии жюри

ТЗ трека дословно: **«практичность, скорость обнаружения, объяснимость решений»** + риск-оценка/приоритизация.

- **Практичность** → готовая очередь инцидентов для SOC + черновик IR-отчёта (экономит ручную работу),
  привязка к реальной боли АФМ/гос-органов РК (утечка 16,3 млн = инсайдер, а не взлом).
- **Скорость обнаружения** → детект в реальном времени по потоку access-логов, мгновенный risk-score
  без обучения и инфраструктуры.
- **Объяснимость** → SHAP-подобная декомпозиция «почему помечено» относительно baseline + граф доступа
  + RU-нарратив IR-отчёта. Центр продукта.
- **Приоритизация** → P1–P4, очередь по score, recall@top-N.
- **Язык метрик** (важно сказать жюри): **НЕ accuracy** — инсайдер-события редкие, классификатор-пустышка
  даёт ~99.9% accuracy и ловит ноль. На верхней панели: **AUPRC** (ранжирование) + **recall@top-N**
  (все реальные инциденты в топе) + **снижение ложных тревог** против наивного объёмного порога.
  Два контрольных negative (`INC-9001`, `INC-9002`) остаются ниже порога — доказывают precision.
  Ползунок порога тревоги (what-if) пересчитывает confusion-matrix вживую — жюри двигает сам.

## Runtime-статус (честно)

**Проверено в браузере jun20** после фикса Babel-runtime. Свежий Babel standalone по умолчанию
транспилирует JSX в automatic runtime и эмитит `import {jsx} from "react/jsx-runtime"`, который
инжектится как classic `<script>` → `SyntaxError: Cannot use import statement outside a module`,
и React не стартует. Фикс — регистрация пресета с **classic runtime** (`React.createElement`,
см. `<script>` в `<head>`), версионно-независимо.

После фикса в браузере: React рендерится, **9 инцидентов** в очереди, верхний — массовая выгрузка
80k (P1), граф-SVG рисует цепочку бокового перемещения, SHAP-панель и переключение инцидентов
работают, кнопка IR генерит черновик, ползунок порога пересчитывает confusion-matrix + AUPRC.
Штатные консольные сообщения — предупреждения Tailwind/Babel CDN («не для продакшена»), это не ошибки.

## Assumptions / честные ограничения

- Детект-«модель» — **детерминированные UEBA-эвристики + взвешенный scoring**, а не обученный
  XGBoost/IsolationForest/autoencoder. Осознанный выбор для 24ч-прототипа: эвристики + граф +
  объяснимость стабильнее и убедительнее, чем недо-обученная модель под дедлайн. На питче честно:
  «scoring калиброван, не обучен; для прод — supervised baseline + autoencoder на реальных
  access-логах (Splunk/ELK/SIEM-экспорт)».
- SHAP здесь — **SHAP-подобная** additive-декомпозиция весов триггеров, а не вызов библиотеки `shap`.
  Семантика та же (вклад фактора в решение), реализация — без Python-рантайма.
- Граф-эвристики работают на маленьких подграфах инцидентов (демо). На потоке миллионов событий
  нужен тот же подход на бэкенде (план описан, в прототип не входит).
- Baseline в прототипе задан вручную на синтетике. В проде baseline обучается на исторических
  логах самого пользователя (скользящее окно), что и есть суть UEBA.
- Данные синтетические и подобраны так, чтобы каждая типология чисто срабатывала (для демо).
  Negative-контроли (ETL, аудит) добавлены специально, чтобы не было «детектора-перестраховщика».
- LLM-нарратив по умолчанию — mock; реальный Claude — опционально по ключу.
