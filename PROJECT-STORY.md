# Insider Guard — вся история проекта (от кода до меня)
*Читается с телефона. Каждый блок = то, что можно рассказать вслух. Всё на честных фактах — чтобы не словить оверклейм.*

---

## 0. В одном абзаце (если совсем нет времени)
Insider Guard — система, которая ловит инсайдера с легальным доступом — того, кого периметровые DLP и SIEM не видят, потому что доступ-то законный. Мы смотрим не на сигнатуры и не на периметр, а на **поведение**: считаем личную норму каждого пользователя и ловим отклонение — массовую выгрузку, боковое перемещение, ночную активность — **до того как данные уйдут**. Каждое решение объяснимо до балла, а инцидент запечатывается в досье, пригодное для следствия. On-prem, на периметре заказчика.

---

## 1. Проблема — почему это вообще нужно
В июне 2025 в Казахстане утекло **16,3 млн записей** персональных данных — порядка 80% населения. По предварительной версии Минцифры это **не взлом, а доступ через легальные учётные записи**: инсайдер либо скомпрометированные креды. Кто-то вошёл «через парадную дверь со своим ключом».

Почему это не ловят существующие средства: **DLP смотрит на периметр, SIEM — на сигнатуры**. А тут доступ легальный, сигнатуры нет — для них это невидимо. При этом инсайдер — самый дорогой вектор (**$4,92M** за инцидент, IBM 2025), и утечки в среднем выявляют **241 день**. Плюс закон: **№94-V** требует, чтобы ПДн граждан РК не покидали страну (локализация с 8 января 2025) — облачные решения сюда не подходят по закону.

---

## 2. Что мы построили (продукт)
Рабочий веб-сервис: SOC-консоль аналитика. Поток:
1. **Очередь** — все инциденты отсортированы по risk-score, опаснейшее сверху.
2. **Инцидент** — карточка: персональный baseline, граф attack-path (цепочка хостов до БД), разбор «почему помечено» (вклад каждого фактора), уровни реагирования.
3. **Валидация** — метрики на ground-truth + честная оговорка про синтетику.
4. **Загрузи свой лог (CSV)** — движок считает baseline из загруженных данных, а не зашит под демо.

Живёт на `insider-guard.vercel.app`. Данные — **только синтетика, 0 реальных ИИН**.

---

## 3. Как это устроено внутри (код/архитектура) — честно
**Стек.** Фронт — один файл на React (через Babel в браузере, без сборки); граф, разбор вкладов, Isolation Forest — vanilla JS на клиенте. Бэк — **Node.js + Express**: ядро `engine.js`, security-заголовки (helmet) + rate-limit, загрузка CSV через `/api/ingest`. Деплой — Vercel (+ Railway для локальной LLM). Ноль внешних/облачных API.

**Логика принятия решения — «как событие становится баллом»:**
- Считаем **личную норму** пользователя из его истории методом *leave-one-day-out* (оцениваемый день исключён из своей же нормы), trimmed-mean объёма, рабочие часы по перцентилям. Если истории мало — падаем на медиану по роли (коллегам).
- Прогоняем **батарею триггеров** с экспертными весами: массовая выгрузка, боковое перемещение, широкий доступ, ночная активность, эскалация привилегий, индикаторы компрометации и т.д.
- Сумму весов **насыщаем** в шкалу 0–100, раскладываем по триггерам так, что **сумма вкладов точно равна баллу, бит-в-бит** (glass-box).

**Главная правда архитектуры (говори уверенно):** балл считает **детерминированный объяснимый движок** — это аудируемость и ноль галлюцинаций. А **Isolation Forest** (своя unsupervised-модель, обучена с нуля на ~980 фоновых user-day, на устройстве) и **robust-MAD** — это **две независимые модели рядом**, которые подтверждают аномалию, но вердикт не считают. AI намеренно убран из источника решения — там, где он опасен. Локальная LLM (Ollama) только пишет черновик отчёта и тоже балл не считает.

**Wow-механики:** граф kill-chain собирается во времени; honeytoken-приманка (касание → ноль ложных по построению); маппинг на MITRE ATT&CK; запечатывание досье в SHA-256 (меняешь байт → INTEGRITY BROKEN); append-only журнал с hash-chain.

## 4. Почему это сильно (ценность + почему только мы)
1. **Превенция, а не вскрытие** — ловим на боковом перемещении, до выгрузки. Конкуренты разбирают после.
2. **Скорость = деньги** — 241 день → секунды. Каждый день промедления = больше выгруженных строк = больше ущерб.
3. **Объяснимо + досье для суда** — каждое решение разложено, инцидент запечатан. Black-box ML юридически пригодного доказательства не даст.
4. **Покупаемо гос-заказчику** — on-prem, №94-V: SaaS-конкурента по закону нельзя, нас можно.

**Moat в одной фразе:** «Black-box ML не даст досье для суда; SaaS не пройдёт по №94-V; DLP/SIEM не видят легальный доступ. Мы закрываем все три.»

## 5. Доказательство + честность (где мы MVP)
На контролируемом синтетическом корпусе: recall@top-8 = 100%, AUPRC = 1.0, precision 100%. И сразу честно: **на синтетике метрики завышены**. Почему вообще не accuracy — на редком событии пустышка «всё чисто» даёт 99,9% accuracy и ловит ноль; поэтому показываем recall@topN и AUPRC, а не accuracy.

**Честная граница (не переступать):** движок реальный и считает baseline из загруженных данных, но веса и порог **калиброваны на синтетике**. На реальном «грязном» логе цифры будут ниже, нужна **рекалибровка в пилоте**. Это MVP, а не продакшен. Эта честность жюри засчитывает как зрелость; пойманный оверклейм — убивает.

## 6. Масштаб + видение
**Масштаб:** атом скоринга — stateless-функция на одного пользователя за день, **шардится горизонтально**, без GPU. Сейчас — батч (лог за секунды). В прод выходим заменой **инфраструктуры, не алгоритма**: in-memory → Postgres, загрузка файла → коннектор к БД аудита (опрос или CDC), baseline → инкрементальный. Новый источник (AD / Colvir / ЦФТ / 1С) — адаптер ~50 строк, не новый продукт.

**Видение:** сейчас MVP → пилот с рекалибровкой → real-time-мониторинг через подключение к БД → **поведенческий слой защиты на всех access-логах организации** (не только выгрузка из БД) → стать **стандартом объяснимой инсайдерской защиты для регулируемых секторов** (финансы, госсектор) в РК и других рынках с локализацией данных. Облако туда не зайдёт по закону, black-box — по объяснимости.
