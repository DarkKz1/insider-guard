# Samples — Insider Guard

Готовые синтетические логи доступа, которые судья может **скачать и загрузить** во фронт
(или прогнать через серверный адаптер `POST /api/ingest`). Движок сам считает per-user
baseline (leave-one-day-out) и поднимает аномалию в топ очереди — без предразметки.

> ⚠️ **Это 100% синтетика. НОЛЬ реальных ИИН и персональных данных.**
> Имена пользователей, ресурсы, IP и хосты — вымышленные.

---

## `insider-access-log.csv`

Реалистичный access-лог оператора данных за ~11 дней (июнь 2026), 98 событий, 3 пользователя.

### Формат колонок (заголовок)

```
user,role,resource,action,rows,ts,host,ip
```

| Колонка    | Тип     | Назначение                                                              |
|------------|---------|-------------------------------------------------------------------------|
| `user`     | string  | Учётная запись (актор).                                                  |
| `role`     | string  | Роль (analyst / etl-service / support) — для сравнения с пирами.        |
| `resource` | string  | Целевой ресурс `DB.table`. Пусто допустимо **только** для `action=LOGIN`.|
| `action`   | enum    | `LOGIN`, `SELECT`, `EXPORT`, `INSERT`, … (см. `server/ingest.js ACTIONS`).|
| `rows`     | int     | Объём затронутых строк — главный объёмный сигнал.                        |
| `ts`       | ISO ts  | `YYYY-MM-DDTHH:MM:SS` (час суток → off-hours-сигнал).                    |
| `host`     | string  | Рабочая станция / узел.                                                  |
| `ip`       | string  | Source IP.                                                              |

Адаптер `server/ingest.js` принимает алиасы заголовков (`username`/`actor`, `table`/`target`,
`timestamp`/`time`, `src_ip` и т.д.) и форматы CSV / JSONL / JSON-array — подключение нового
источника РК (AD/WEF, Colvir/ЦФТ, СУБД-аудит, 1С) ≈ 50 строк маппинга.

### Что внутри (ground truth для демо)

| Пользователь       | Роль        | Профиль                                                        | Ожидаемый вердикт |
|--------------------|-------------|----------------------------------------------------------------|-------------------|
| `aibek.analyst`    | analyst     | ~50 строк/день, рабочие часы 09–18, 10 дней истории… **+ один аномальный день: ночная выгрузка 62 000 строк PII-таблицы `DB_CUSTOMERS.persons` в 02:47–02:49 (SELECT+EXPORT)**. | 🚨 **P1 Критический, score 94** |
| `svc_etl_nightly`  | etl-service | Легитимный ночной ETL: стабильно ~840 000 строк каждую ночь в 01:00. | ✅ **НЕ помечен** (постоянная норма — leave-one-day-out не видит отклонения). |
| `dana.support`     | support     | Обычный support: мелкие дневные lookups по тикетам.            | ✅ **НЕ помечен** |

Ключевой контраст для жюри: **большой объём сам по себе ≠ угроза**. Ночной ETL грузит
в 13× больше строк, чем пиковая выгрузка аналитика, но он легитимен, потому что это
**его личная норма**. Аналитик ловится не по абсолютному объёму, а по отклонению от
**собственного** baseline (×N от своей нормы и часов) — off-hours + volume-spike +
bulk-export. SHAP-разложение: `OFF_HOURS 32 + VOLUME 25 + BULK_EXFIL 21 + STAGING_EXFIL 16 = 94`
(Σ === score, точно).

### Как загрузить

**Через фронт:** перетащить `insider-access-log.csv` в зону загрузки.

**Через серверный адаптер:**
```bash
curl -X POST http://localhost:3000/api/ingest \
  -F "file=@samples/insider-access-log.csv" \
  -F "name=Judge Sample - Insider Access Log"
```
Ответ: `datasetId`, `incidentCount`. Затем
`GET /api/incidents?datasetId=<id>` покажет `aibek.analyst` с P1/score 94 в топе.
